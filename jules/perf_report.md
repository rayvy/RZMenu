# Анализ проблем с производительностью и скоростью запуска (RZMenu)

В ходе анализа кодовой базы (особенно директории `qt_editor`) были выявлены следующие основные проблемы, которые являются узким горлышком, вызывающим долгий запуск, фризы и потенциальные краши:

## 1. Блокирующий и избыточный парсинг файловой системы при запуске
Во вкладках `TexWorks` (в частности `TexWorksMainTab`, `TexWorksResourcesTab`, `TexWorksTestTab` и др., расположенных в `qt_editor/widgets/texworks_panel.py` и `qt_editor/widgets/texworks/tabs.py`) выполняется агрессивное сканирование папок (`utils.scan_textures` / `image_utils.scan_textures`) прямо в процессе инициализации виджетов или при первом их обновлении (`refresh` / `QTimer.singleShot(100, self.refresh)`).
* Сканирование может затрагивать огромные директории модов (сотни или тысячи текстур).
* Использование `os.walk` при `recursive=True` сильно бьёт по производительности жесткого диска и задерживает запуск редактора.

## 2. Синхронные блокировки Blender API (Deadlocks / Freezes)
Мьютекс `_blender_lock` в модуле `qt_editor/lib/image_utils.py` используется для сериализации вызовов к Blender API (например, `bpy.data.images.load` и `img.pixels.foreach_get`) из фоновых потоков `ThumbnailWorker` и `AtlasWorker`.
* Blender API **не является потокобезопасным**. Вызов API из сторонних потоков (даже с мьютексом для них самих) может конфликтовать с внутренним циклом обработки событий Blender, что приводит к "заморозке" (freeze) всего приложения или даже крашу.
* Процесс загрузки текстур `load_texture_data` сначала резолвит путь, потом грузит изображение в Blender, извлекает пиксели, конвертирует их в `numpy`, а затем в `QPixmap`. Все это крайне тяжелые операции, которые не должны происходить синхронно или блокировать основной поток. При наличии десятков текстур в `TexWorksMainTab` это полностью вешает поток интерфейса.

## 3. Архитектура обновления интерфейса (Избыточные refresh)
* В `qt_editor/core/launcher.py` есть обработчики `on_depsgraph_update` и `on_undo_redo`, которые дергают `_window.sync_from_blender()` и `_window.full_refresh()`. Депсграф обновляется очень часто при любых манипуляциях, что вызывает каскад обновлений всего интерфейса.
* Метод `full_refresh` в `window.py` делает `SIGNALS.structure_changed.emit()`. Этот сигнал подхвачен почти каждым виджетом (`Outliner`, `Viewport`, `Inspector`, `TexWorksPanel`), заставляя их пересобирать списки и виджеты заново.
* Создание и удаление Qt-виджетов "на лету" (в циклах `while self.layout.count() > 0: widget.deleteLater()`) при каждом чихе вызывает дикую фрагментацию памяти (memory leak) и сильно грузит процессор (overhead на создание объектов QWidget).

## 4. Глобальные сигналы без фильтрации контекста
Как уже упомянуто в `CODE_REVIEW.md`, сигнал `data_changed` является глобальным. Изменение одной переменной триггерит обновление `Outliner`, `Viewport` и `Inspector`. И хотя в `inspector.py` была попытка добавить throttle (`_refresh_timer`), это лишь маскирует проблему, но не решает её в корне. В других виджетах троттлинга часто нет (или он составляет 16-30 мс, что всё равно слишком часто).

## 5. Инициализация Qt Application
При импорте модуля `qt_editor/core/launcher.py` или вызове `get_app()`, запускается `QApplication` с `sys.argv`. Если это происходит во время инициализации аддона (при загрузке Blender), это может замедлить запуск самого Blender. Ожидание Qt event loop внутри таймера Blender (`process_qt_events` каждую 0.01 сек) создает постоянную нагрузку на CPU даже в состоянии простоя.

## Возможные пути решения (Технический долг)
1. **Ленивая загрузка (Lazy Loading):** Избегать вызова `refresh()` или сканирования файловой системы до тех пор, пока пользователь физически не откроет вкладку `TexWorks`.
2. **Кэширование путей и метаданных:** Сканировать файловую систему в фоне (нативных потоках, не трогая Blender API), а результаты хранить в легковесном JSON-кэше, к которому UI обращается мгновенно.
3. **Обновление только видимых элементов (Virtualization):** Вместо пересоздания сотен `QWidget` в скролл-зонах использовать `QListView` с кастомными делегатами или пул переиспользуемых виджетов (Widget Pooling).
4. **Асинхронные задачи Blender:** Вместо блокировки мьютексом использовать `bpy.app.timers` для добавления задач по загрузке пикселей изображений в очередь главного потока (Main Thread), чтобы Blender сам их обрабатывал в безопасном контексте.
