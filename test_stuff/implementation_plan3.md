# Честный план: Live2D — починка данных и UI

## Что реально есть и что сломано

### Данные (Python + бинарный экспорт)
- `blender_props.py` — структуры данных костей, keyframe, mesh, weights **есть**, работают.
- `exporter.py` — `pack_live2d_data()` **есть**, вызывается из `j2_exporter.py`.
- **Критическая проблема 1**: бинарники пишутся напрямую в `export_path` (например, `C:\mod\bones.bin`), тогда как в `core.j2` они указаны как `.\res\bones.bin`. Папка `res/` **никогда не создаётся** — файлы падают с `FileNotFoundError`, который проглатывается блоком `except Exception` и выводится лишь в консоль. Buffers never appear.
- **Критическая проблема 2**: даже если костей нет — пустышки не пишутся. 3DMigoto не стартует без файлов, на которые указаны `filename = .\res\*.bin`.
- `animation_system.py` — **полностью готовый** класс для экспорта анимаций в binary, но он **нигде не вызывается** и `animations.bin` вообще не фигурирует в `core.j2`.

### Qt Editor
- **Таймлайна нет** — папка `graphical_editor/` пустая.
- **Редактора костей нет** — только поле `Mesh Offset` в инспекторе.
- Данные в Blender (`rzm.skeleton.bones`, `rzm.skeleton.animations`) есть — их можно читать и писать через операторы.

### Шейдеры / templates
- `animate_bones.hlsl`, `draw_instancer.hlsl`, `core.j2` — в целом верны, но не тестировались т.к. `.bin` не пишутся.
- `ResourceAnimatedBoneBuffer` привязываетс как `vs-t109`, хотя в `draw_instancer.hlsl` он читается как `t109` — регистры совпадают.

## Приоритеты (по очерёдности, а не горизонтально)

### Фаза 1 — Починка экспорта буферов (ДАННЫЕ)

**Цель**: при любом экспорте (Quick Update, полный экспорт) файлы `res/bones.bin`, `res/mesh.bin`, `res/weights.bin` **всегда записываются**, даже пустышки минимального размера.

> [!IMPORTANT]
> **Ключевое изменение**: путь записи меняется с `export_path/bones.bin` на `export_path/res/bones.bin`, и папка создаётся автоматически.

**Файлы:**
- `core/live2d/exporter.py` — исправить путь на `os.path.join(export_path, "res", "*.bin")`, добавить `os.makedirs(..., exist_ok=True)`, добавить запись пустышек если нет данных.

---

### Фаза 2 — Базовый UI костей в Configurator (не в Inspector)

**Цель**: вкладка **"Skeleton"** в `RZConfiguratorManager` с возможностью:
- Увидеть список костей (имя, родитель, позиция).
- Добавить / удалить кость.
- Задать имя скелета.

> [!NOTE]
> Кости — глобальные (`rzm.skeleton`), не привязаны к конкретному LIVE2D-элементу. Это правильный дизайн: один скелет на проект (как в Live2D Cubism — один риг).

**Файлы:**
- `qt_editor/widgets/configurator.py` — добавить `SkeletonTab` и зарегистрировать её в `RZConfiguratorManager.add_tab()`.
- Новые Blender-операторы в `operators/element_ops.py` или отдельный файл — `rzm.add_bone`, `rzm.remove_bone`.

---

### Фаза 3 — Базовый таймлайн (минимальный, функциональный)

**Цель**: в той же вкладке "Skeleton" или отдельной вкладке "Animation" — список анимаций + список keyframe для выбранной кости выбранной анимации.

> [!NOTE]
> **Полноценный timeline-виджет** (с полосой времени, перетаскиванием keyframe) — это значительно больше работы. На этом этапе делаем список-форму для управления данными. Визуальный таймлайн — следующая итерация, после подтверждения что данные правильно экспортируются.

**Файлы:**
- `qt_editor/widgets/configurator.py` — `AnimationTab` с QListWidget для анимаций и простой формой для keyframe.
- Blender-операторы: `rzm.add_animation`, `rzm.add_keyframe`, `rzm.remove_keyframe`.
