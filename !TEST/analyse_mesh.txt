## Точная механика поломки

**Сценарий: объект с applied Mirror+Merge**

1. `.id` атрибута **нет** (мы это уже доказали) → `has_id = False`
2. `eval_v_count = orig_v_count` (модификаторов нет, меш финальный) → `eval_v_count > orig_v_count` = **False**
3. `use_eval_path = (True and False)` = **False**
4. Идёт в **ORIG mode**

ORIG mode делает:
```python
sk_co = np.empty(orig_v_count * 3, dtype=np.float32)
sk_blk.data.foreach_get('co', sk_co)
deltas_all = sk_co - ba_co  # дельты по индексу
```

Затем применяет:
```python
buf_f32[vb_off + i] += deltas_all[v_map[i]]
```

`v_map[i]` — это eval vertex index из signature matching. После merge индексы переупакованы. `deltas_all[v_map[i]]` читает дельту по **новому индексу**, но шейпкей хранит дельты по **старому порядку** оригинальной половины меша.

## Где именно фикс

В `export_cache.py`, `build_cache_from_efmi`, когда строится кэш для объекта **без `.id`** и **без активных модификаторов** (applied меш):

```python
result = reconstruct_vertex_map_from_mesh(eval_mesh, efmi_obj, stride, flip_winding)
v_map_topology, eval_v_count, has_id = result if result else (None, 0, False)
```

`has_id=False` и `eval_v_count == orig_v_count` → puppet_master выбирает ORIG mode → читает дельты по неправильным индексам.

**Фикс**: когда `.id` отсутствует и меш уже applied — нужно принудительно форсировать EVAL mode, потому что после merge индексы нестабильны и ORIG mode неверен.

```python
# В build_cache_from_efmi, после получения result:
# Если меш applied (нет .id) но orig_v_count != "чистая" половина
# — пометить has_real_id=False И установить eval_v_count > orig_v_count искусственно
# чтобы puppet_master пошёл в EVAL mode
```

Но EVAL mode тоже проблемный для applied меша — там нет модификаторов, depsgraph не даст Mirror геометрию.

## Реальное решение

После merge объект **уже является финальным мешем**. Нет зеркала, нет модификаторов. Puppet master должен читать дельты **напрямую из `mesh.shape_keys`** но через правильный маппинг индексов.

`v_map` из signature matching уже содержит правильный маппинг `buf_slot → eval_vert_idx`. Проблема в том что ORIG mode использует `v_map[i]` как индекс в шейпкее, а шейпкей после merge имеет **другой порядок** чем eval меш.

Для applied меша eval == data (нет модификаторов), поэтому `sk_blk.data[v_map[i]]` должно работать корректно — `v_map[i]` это eval индекс который совпадает с data индексом.

Нужно проверить одну вещь: что именно содержит `v_map` для applied объекта — индексы в диапазоне `[0, eval_v_count)` или что-то другое. Можешь добавить один принт в `_parity_map_from_triangulated` перед return:

```python
print(f"DEBUG v_map sample: {results[:10]}, eval_v_count={eval_v_count}, has_id={has_id}")
```

И запустить экспорт на проблемном объекте. Это покажет точно что лежит в `v_map` и подтвердит где именно разрыв.


Ключевое место в `_bake_with_direct_offsets`, условие роутинга:

```python
use_eval_path = (not has_real_id and eval_v_count > orig_v_count)
```

**Проблема:** после Mirror+Merge+apply — `has_real_id=False`, но `eval_v_count == orig_v_count` (модификаторов нет, меш финальный). Условие даёт `False` → идёт в ORIG mode → читает дельты по индексам которые переупакованы merge'ем.

Но EVAL mode тоже не поможет для applied меша — там нет модификаторов, depsgraph вернёт тот же меш.

**Реальный фикс — в export_cache.py.** Когда `.id` отсутствует и меш уже applied, нужно пометить `has_real_id=True` принудительно — потому что для applied меша `eval == data`, индексы совпадают, и ORIG mode корректен **если** `v_map` содержит правильные индексы.

Вопрос: `v_map` из signature matching для applied меша — он корректен? После merge индексы переупакованы, но signature matching строит маппинг `buf_slot → eval_vert_idx` через UV/Color сигнатуры, что не зависит от порядка индексов. Значит `v_map[i]` указывает на правильную вершину в eval меше.

Для applied меша `eval == data`, поэтому `sk_blk.data[v_map[i]]` должно дать правильную дельту.

Так что фикс минимальный — в `export_cache.py`, `build_cache_from_efmi`, после получения result:

```python
result = reconstruct_vertex_map_from_mesh(eval_mesh, efmi_obj, stride, flip_winding)
v_map_topology, eval_v_count, has_id = result if result else (None, 0, False)

# ФИКС: для applied меша (нет модификаторов, нет .id) eval == data.
# Signature v_map уже корректен. Помечаем has_real_id=True чтобы
# puppet_master использовал ORIG mode с правильным v_map.
if not has_id and not efmi_obj.modifiers:
    has_id = True
```
